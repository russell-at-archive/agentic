# Task Tracking Process

## Philosophy

Task tracking is the operational layer that sits between planning and
delivery. The planning process produces a sequenced, dependency-ordered
`tasks.md`. Task tracking governs how those tasks move from definition to
completion, who owns them, how blockers surface, and how progress is visible
to the team.

Tracking is not a status-reporting ceremony. It is a coordination mechanism
that prevents duplicate work, surfaces blockers early, and creates a
continuous record of where a feature is in its lifecycle. A well-tracked
task tells the next person who picks it up exactly what has happened, what
remains, and what constraints they are working under.

AI agents change the tracking calculus. An agent executing a task must leave
an auditable trail — what was attempted, what succeeded, what failed, and
what the next human action must be. Human developers must be able to resume
from any point in a partially-executed task without re-reading full
conversation history.

## Core Principles

- Every task has exactly one owner at any point in time.
- Status transitions are explicit and timestamped.
- Blockers are escalated immediately, not absorbed silently.
- No task is marked complete unless acceptance criteria are verified.
- Scope changes require returning to planning, not silent expansion.
- Agents leave an auditable trail sufficient for a human to resume.
- The canonical task state is the GitHub Issue, not a local notes file.

## Task States

Each task moves through a defined set of states. Skipping states is not
allowed. Returning to a prior state when a blocker is discovered is required.

| State | Meaning | Who transitions |
| --- | --- | --- |
| `backlog` | Defined, not yet scheduled | Planner |
| `ready` | Dependencies met, unassigned | Planner or tech lead |
| `in-progress` | Actively being worked | Assignee |
| `blocked` | Cannot proceed without external input | Assignee |
| `in-review` | Pull request open, awaiting review | Assignee |
| `done` | Acceptance criteria verified and merged | Reviewer or assignee |

### State Transition Rules

- `backlog` → `ready`: All task dependencies are merged and verified.
- `ready` → `in-progress`: Assignee claims the task and opens a branch.
- `in-progress` → `blocked`: A blocker is identified and documented.
- `blocked` → `in-progress`: Blocker is resolved and documented.
- `in-progress` → `in-review`: Pull request is opened with spec traceability.
- `in-review` → `in-progress`: Review finds defects; assignee addresses them.
- `in-review` → `done`: Review passes, acceptance criteria verified, merged.

A task may never move from `blocked` to `done` directly. The blocker must be
resolved and the task must re-enter `in-progress` before it can go to review.

## Task Anatomy

Every task, whether tracked in `tasks.md` or a GitHub Issue, must carry the
following fields before it enters `ready` state.

| Field | Required | Description |
| --- | --- | --- |
| Task ID | Yes | Numeric identifier scoped to the feature (`T-01`, `T-02`) |
| Title | Yes | One-line imperative statement of the outcome |
| Objective | Yes | Two to five sentences explaining the purpose |
| Dependencies | Yes | Task IDs that must be merged before this task can begin |
| Acceptance criteria | Yes | Numbered list of verifiable outcomes |
| Files in scope | Yes | Paths or subsystems likely touched |
| Required tests | Yes | Categories and what each validates |
| Non-goals | Yes | Explicit exclusions to prevent scope creep |
| Assignee | No | Set when the task moves to `in-progress` |
| Branch | No | Set when the task moves to `in-progress` |
| PR link | No | Set when the task moves to `in-review` |
| Blockers | No | Populated when the task moves to `blocked` |

## Tooling

Task tracking uses two surfaces in combination.

### tasks.md (local source of truth during planning)

`tasks.md` is generated by `/speckit.tasks` and lives in
`specs/<###-feature-name>/`. It is the authoritative planning artifact during
task decomposition. It is a flat checklist with structured metadata per task.

`tasks.md` is read-only after `/speckit.analyze` passes and tasks are pushed
to GitHub Issues. Do not update `tasks.md` to reflect execution progress;
GitHub Issues carry live state.

### GitHub Issues (live state during execution)

Run `/speckit.taskstoissues` to create one GitHub Issue per task, ordered by
dependency. Issues are the canonical location for:

- current status (label-based)
- assignee
- branch and PR links
- blocker notes
- execution log from agents

Label taxonomy:

| Label | Meaning |
| --- | --- |
| `status:backlog` | Defined, dependencies not yet met |
| `status:ready` | Dependencies met, unassigned |
| `status:in-progress` | Actively being worked |
| `status:blocked` | Blocked; blocker documented in issue body |
| `status:in-review` | PR open |
| `status:done` | Merged and verified |
| `type:feature` | New user-visible behavior |
| `type:bug` | Defect fix |
| `type:refactor` | Internal restructuring, no behavior change |
| `type:chore` | Dependency, tooling, or infrastructure work |
| `priority:p1` | MVP; must ship before any P2 work begins |
| `priority:p2` | High value; ships after MVP is verified |
| `priority:p3` | Nice to have; ships after P2 is verified |

## Claiming a Task

Before starting work on a task:

1. Confirm all dependency tasks carry `status:done`.
2. Assign yourself to the GitHub Issue.
3. Replace the `status:ready` label with `status:in-progress`.
4. Open a branch named `<task-id>-<short-slug>` (e.g., `t-03-add-auth`).
5. Add the branch name to the issue body.

Do not start work without completing all five steps. Untracked in-progress
work is invisible to the team and to agents resuming the work.

## Reporting Blockers

When a task cannot proceed:

1. Replace `status:in-progress` with `status:blocked`.
2. Add a blocker comment to the issue with:
   - What is blocking progress (concrete description, not "waiting on X")
   - What is needed to unblock (owner, decision, information)
   - Who has been notified
   - Date the blocker was identified
3. Do not continue work on the blocked task. Do not silently accumulate
   partial changes.
4. When the blocker is resolved, add a resolution comment documenting what
   changed, then return the task to `status:in-progress`.

## Opening a Pull Request

When a task is ready for review:

1. Open a pull request from the task branch to the main integration branch.
2. The PR description must include:
   - Link to the GitHub Issue
   - Link to `spec.md` and the relevant acceptance criteria
   - Link to the specific task in `tasks.md`
   - Summary of what was implemented
   - Test evidence (test run output or screenshot)
   - Any deviations from the plan and the justification
3. Replace `status:in-progress` with `status:in-review`.
4. Add the PR link to the issue body.

A PR that cannot be traced back to an approved spec and plan task will be
returned for clarification before review begins.

## Closing a Task

A task is `done` only when:

- The PR is merged to the integration branch.
- All acceptance criteria in the issue are checked off.
- Required tests are present and passing in CI.
- The reviewer has explicitly approved traceability (spec → plan → task → PR).

Replace `status:in-review` with `status:done` after merge. Do not close the
issue until all checks are met.

## Agent Execution Protocol

When an AI agent is executing a task, it must maintain a running log in the
GitHub Issue body under an `## Execution Log` section. Each log entry
contains:

- timestamp
- action taken
- outcome (success, failure, partial)
- next step or handoff instruction

The execution log must be sufficient for a human to resume the task from any
point without re-reading the agent session. An agent that completes a task
must update issue state identically to a human developer.

If an agent encounters a blocker:

1. It documents the blocker in the issue (same format as human blocker
   reporting).
2. It transitions the issue to `status:blocked`.
3. It stops work and surfaces the blocker to a human reviewer.
4. It does not attempt to resolve architectural or product ambiguity
   autonomously.

## Scope Changes During Execution

If implementation reveals that a task requires changes beyond its defined
scope:

1. Stop. Do not expand scope silently.
2. Document the discovery in the issue.
3. Return to planning: amend `plan.md` and re-run `/speckit.tasks` if the
   change is significant.
4. Create a new task for the expanded scope.
5. Resume the original task at its original scope.

Scope expansion absorbed into an existing task is a planning failure, not a
delivery success.

## Progress Visibility

The team tracks feature-level progress through the GitHub Issues board
filtered by feature label. Individual task progress is visible through issue
labels and comments.

A feature is considered complete when:

- All tasks for that feature carry `status:done`.
- The MVP acceptance scenario in `quickstart.md` has been verified end to end.
- The feature branch is merged and the `specs/<###>/` artifacts are archived.

## Integration with the Planning Lifecycle

Task tracking integrates with the planning process at two points.

**Entry point**: After `/speckit.analyze` passes at the end of Phase 4 (Task
Decomposition), run `/speckit.taskstoissues` to create GitHub Issues. This
is the moment tracking begins.

**Exit point**: When all issues for a feature carry `status:done` and the
end-to-end acceptance scenario passes, the feature is complete and the
planning artifacts may be archived.

No task may enter `in-progress` before gates 1 through 5 of the planning
lifecycle are satisfied. No task may be marked `done` without verified
acceptance criteria.

## Gate Rules

| Gate | Condition |
| --- | --- |
| T-1 | `/speckit.analyze` passes before any issue is created |
| T-2 | All dependency tasks are `done` before a task enters `ready` |
| T-3 | Branch opened and issue updated before work begins |
| T-4 | Blocker formally documented before task enters `blocked` |
| T-5 | PR traces to spec, plan, and task before review begins |
| T-6 | All acceptance criteria verified before task is `done` |

## Risks and Mitigations

| Risk | Likelihood | Mitigation |
| --- | --- | --- |
| Tasks marked done before acceptance criteria are verified | High | Gate T-6 requires explicit checklist sign-off; reviewer owns verification |
| Agent expands scope silently | High | Agent protocol requires stopping and documenting when scope is exceeded |
| Blockers absorbed rather than surfaced | Medium | Blocker state is a required label; work cannot continue on a blocked task |
| Issues become stale while work continues in local branches | Medium | PR link in issue body is required before review begins |
| Dependency ordering violated | Medium | Gate T-2 enforces dependency state before task enters ready |
| Tasks bypass planning when spec changes | Low | Scope changes during execution trigger a planning return before new tasks begin |

## Open Questions

- Should sprint cadence or kanban flow govern task scheduling?
- Who has authority to transition a task to `done` — assignee, reviewer, or
  both?
- Should the execution log format for agents be enforced via an issue template
  or left as convention?
- At what task volume does a GitHub Project board replace label-based
  filtering?
- Should `tasks.md` be updated to reflect done state post-merge, or is
  GitHub Issues the only live state?
