# Unified Task Tracking Process (UTTP)

This document defines the unified standard for task tracking in the Archive Agentic workflow. It merges the operational rigor of Claude, the managerial insights of Codex, and the developer-centric agility of Gemini into a single, cohesive process.

## Philosophy

Task tracking is the operational bridge between planning and delivery. We treat tracking not as a status-reporting ceremony, but as a coordination mechanism that ensures:

1. **Traceability**: Every line of code can be traced back to an approved task, plan, and specification.
2. **Auditability**: Agents and humans leave a clear trail of intent, action, and outcome.
3. **Flow**: Work moves in thin, vertical slices with minimal context switching and no silent scope expansion.

## Core Principles

- **Dual-Sync Truth**: GitHub Issues are the canonical record of truth; `tasks.md` is the local operational cache.
- **Agent Accountability**: Agents must maintain an execution log that allows a human to resume their work at any point.
- **Evidence-Based Done**: A task is only "Done" when it is merged and verifiable evidence (tests, logs, or screenshots) is attached.
- **WIP Discipline**: Limit work-in-progress to one active task per owner to maximize focus and throughput.

## The Dual-Sync Model

We use two surfaces in combination to balance speed and visibility.

### 1. Local: `tasks.md`

Located in `specs/<feature-name>/tasks.md`, this file is the developer's scratchpad. It is generated by `/speckit.tasks` and serves as the immediate sequence of work.

- Use simple markers: `[ ]` (Not Started), `[/]` (In Progress), `[x]` (Completed), `[~]` (Blocked).
- This file is highly volatile during a single session but must be synced to GitHub before the session ends.

### 2. Remote: GitHub Issues

GitHub Issues provide the team's canonical visibility. They are created from `tasks.md` via `/speckit.taskstoissues`.

- Issues house the **Execution Log**, **Assignee**, **Branch Links**, and **Blocker Notes**.
- Issues are the authoritative source for project boards and milestone tracking.

## State Model & Transition Rules

We use a shared set of durable states. Skipping states or moving backward without documentation is not permitted.

| State | Meaning | Transition Condition |
| :--- | :--- | :--- |
| `backlog` | Defined in `tasks.md`, not yet an issue. | Created via `/speckit.taskstoissues`. |
| `ready` | Issue exists, dependencies are `done`. | All upstream Task IDs are marked `done`. |
| `in-progress` | Owner assigned, branch opened. | Assignee claims the task and updates the state. |
| `blocked` | Progress halted by external factor. | Blocker is documented with owner and next step. |
| `in-review` | Pull Request open, awaiting validation. | PR link and validation evidence are attached. |
| `done` | Merged and verified against AC. | PR is merged; all acceptance criteria are checked. |

## Agent Execution Protocol

When an AI agent executes a task, it must maintain a running log in the GitHub Issue body under a `## Execution Log` section.

### Log Entry Format

```text
[TIMESTAMP] - [ACTION] - [OUTCOME]
Next Step: [INSTRUCTION]
```

- **Autonomy Boundary**: If an agent encounters a decision not covered by the `plan.md` or `spec.md`, it MUST transition the task to `blocked` and stop. Agents do not "hallucinate" architectural or product decisions.
- **Session Handoff**: Before closing a session, the agent must update the Issue with the current state of the branch and any remaining sub-tasks.

## WIP Limits and Operational Metrics

To maintain high velocity and low drift, we enforce the following:

- **WIP Limit**: 1 `in-progress` task per owner (human or agent).
- **Stale Warning**: Any task in `in-progress` for more than 2 working days without a log update is considered "Stale" and requires immediate review.
- **Blocker Escalation**: Blockers older than 24 hours must be escalated to the tech lead for resolution.

## Completion Evidence

A task cannot move to `done` without **Validation Evidence**. This is the concrete proof that the code satisfies the implementation plan. Required evidence types include:

1. **Automated Test Results**: CI logs or local test suite output.
2. **Manual Verification**: Screenshots or recordings for UI changes.
3. **Contract Validation**: Logs showing successful API or schema interactions.

## Scope Changes

If implementation reveals that a task requires work beyond its defined scope:

1. **Stop Implementation**: Do not expand scope silently.
2. **Update Planning**: Amend the `plan.md` and re-run `/speckit.tasks`.
3. **Sync Issues**: Re-run `/speckit.taskstoissues` to reflect the new tasks.
4. **Resume**: Continue the original task at its original scope or move to the new task.
